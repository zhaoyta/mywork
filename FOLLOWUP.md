## 系统负责人补充说明

### 1. 如果这套系统后续由你负责

**当前可用的部分：**

- OCR 解析管道（PaddleOCR 主路径 + Tesseract 降级）在 GB/T 1568-2008 这份标准文件上跑通，原始识别文本缓存在 `data/ocr_cache/`，可人工交叉比对；
- ChromaDB 向量库 + `text-embedding-3-small` 嵌入，分块策略（`chunk_size=500 / overlap=100`）已经过调参；
- RAG 问答链路（检索 top_k=5 + LLM 生成）在 11 道测试题上验证过，自检（`self_check.py`）对 supported / uncertain / unsupported 三态的判断稳定；
- 评估脚本 `tests/evaluate.py` 支持基线对比，可作为回归门禁。

**还不能放心交付的部分：**

- **跨页表格**：PaddleOCR 按页独立处理，跨页合并单元格会丢失结构，T01/T02 类问题在真实多表格文档上存在失准风险；
- **多文档管理**：当前知识库绑定单一 PDF，向量库没有 `source` 过滤，多文档混入会导致检索污染；
- **数据权限**：没有任何访问控制，不适合有多租户或信息分级需求的客户；
- **评估覆盖率**：11 道测试题针对单份 GB 标准设计，不能代表客户实际文档的多样性。

**下一步优先处理（按价值排序）：**

1. **多类型文档接入**：在 `pdf_parser.py` 之上抽象 `DocumentParser` 接口，接入 Word/HTML/Markdown/图片，打通多渠道（邮件正文、会议纪要、Wiki 页面、聊天记录摘要）的清洗与入库管道，这是扩大客户适用面的核心；
2. **评估体系建设**：建立离线三层实验平台——embedding 质量评估（MRR/nDCG@k）、检索链路评估（召回率/精确率）、端到端 RAG 评估（答案命中率 + 自检通过率）。三层可以独立跑，便于定位是哪个环节出了问题；
3. **数据安全体系**：按信息敏感等级对文档分级（公开 / 内部 / 机密），在向量库 chunk 写入时打 tag，检索时通过 `where` 条件过滤，实现基于数据分级的权限控制；
4. **向量库优化**：基于真实业务语料对比不同 embedding 维度（768 vs 1536 vs 3072）的检索效果和内存占用，决定是否切换到本地 bge 或 BGE-M3；大规模场景考虑 HNSW 索引参数和分片策略。

### 2. 如果要给客户试用

试用前我会和产品、客户、技术负责人逐一对齐以下问题：

- **文档类型和来源**：客户会提供哪些格式（扫描版 PDF、Word、HTML、系统导出）？是否含图表/图纸/合并单元格表格？这决定解析策略的选型；
- **信息安全要求**：文档有没有密级划分？向量库部署在客户私有环境还是云端？是否需要单租户隔离？这是能不能上线的硬约束；
- **问答场景边界**：是内部知识检索还是对外服务？允许"不知道"拒答还是要求尽量给出答案？——影响 self_check 阈值的设定；
- **验收标准**：第一版的通过标准是什么？是关键词命中率 ≥ 80%，还是人工抽查 20 道题通过 16 道？明确后才能用 `evaluate.py` 对齐基线；
- **运行环境**：有 GPU 吗？能联网调用 OpenAI 还是必须本地模型？影响 OCR 引擎和 embedding 路径的选择。

---

### 3. 如果要保证质量和稳定性

**验证方式：**

- 每次变更后跑 `uv run python tests/evaluate.py --baseline tests/results/baseline.json`，检查命中率和自检通过率是否退化；
- 人工抽查 `data/ocr_cache/` 中的原始文本，确认 OCR 识别质量（这是整个链路的上游，识别错了后面再好也没用）；
- 对新文档先看 ingest 日志中的 `total/text/table` 分布，异常比例（如 table 占 90%）说明解析有问题，需要提前排查。

**出问题时的处理流程：**

1. **定位层级**：先看 self_check verdict 分布——如果 unsupported 率突然升高，大概率是检索端出了问题（chunk 没命中）；如果 uncertain 率高，大概率是 LLM 输出质量或 prompt 问题；
2. **止损**：self_check 超时已有 fallback（`_FALLBACK = {"verdict": "uncertain", ...}`），10 秒内无响应自动降级，不会挂死；检索端可以把 `score_threshold` 从 0 调高，过滤低置信检索结果；
3. **修复确认**：修改后跑完整评估集，对比修改前后的 per-category 通过率（factual / table / no_answer / ambiguous / ocr_error 各一类），确保不是"修了 A 破了 B"。

### 4. 如果时间有限

我的排优先级原则：**先保住演示/交付的关键路径，再补健壮性，最后才是锦上添花。**

具体做法：

- 列出所有未完成项，对每一项评估：如果没做，评审/演示会不会直接挂掉？会挂掉的优先，不会的排后；
- 跨页表格、多文档这类局限，在 README 或演示中提前说明，而不是临时赶工做半成品；
- 风险同步要早不要晚：如果周四晚评估发现通过率差 5%，周五早上就应该告知，而不是等修完再说——给对方时间调整预期；
- 我自己会保底跑完 `evaluate.py` 确认核心指标，单元测试 `test_unit.py` 作为快速冒烟，不会因为"时间不够"跳过这两步。

### 5. 你会如何使用 AI 或智能体工具

**让 AI 做的事：**

- 模块骨架生成（`pdf_parser.py`、`knowledge_base.py` 的函数签名和基本结构）——能快速给出合理的接口设计；
- Prompt 初稿（self_check prompt、RAG 的 system message）——AI 能给出不错的起点，我来迭代；
- 测试用例的扩充建议——给出已有用例让 AI 提议盲区，但不直接用 AI 生成的"参考答案"；
- 文档和注释初稿。

**必须自己把关的事：**

- **OCR 识别质量**：看 `data/ocr_cache/` 里的原始文本，这是人眼判断，AI 看不到扫描件的实际质量；
- **自检 prompt 的边界行为**：uncertain/unsupported 的触发条件，我会构造边界 case 手动跑，确保没有明显误判；
- **评估测试集的题目设计**：题目要基于 PDF 真实内容，不能让 AI 编题（它会编出文档里没有的内容）；
- **最终评估报告的解读**：数字好看不等于没问题，要看每道题的具体答案，而不只是看通过率百分比。

**如何验证 AI 输出：**代码部分逐函数审阅逻辑（不只是跑通），关注降级路径和异常处理；prompt 输出用边界样本验证 JSON 格式稳定性（`_parse_json` 里已有 markdown fence 的兼容处理，就是因为 GPT 有时会加反引号）。

### 6. 补充一个真实例子

**背景**：这次作业初版 OCR 用 PaddleOCR，但本地 Mac 环境 C++ 编译链有问题，`paddlepaddle` 安装卡住，直接影响核心功能跑通。

**判断**：距离提交还有不到半天，继续排查编译问题代价太高，而 Tesseract 可以通过 brew 几分钟装好。我判断：与其花 2 小时修环境，不如把降级路径做完善，保证两条路都能跑通。

**行动**：在 `pdf_parser.py` 中实现了 `OCR_ENGINE` 环境变量切换，PaddleOCR 作为主路径但捕获 ImportError 自动 fallback 到 Tesseract，README 中写清楚两条启动路径和报错处理方法。同时在 `.env.example` 中加了 `OCR_ENGINE=tesseract` 作为最低依赖启动选项。

**结果**：Tesseract 路径当天完整跑通了演示流程，PaddleOCR 路径留作有完整 C++ 环境时的主路径。评审方可以用任意一条路径复现，不依赖特定环境。

这个决策的核心是：**当排查成本超过绕过成本时，先绕过保住主干，再事后补强**——这也是我面对时间压力时的一贯判断方式。

## 工作方式补充确认

### 1. 遇到不清楚但又比较急的需求

**先确认三件事**：目标是什么（用户要解决什么问题，不是要什么功能）、成功长什么样（最低验收标准）、哪些是硬约束（时间、环境、数据）。这三个问题 10 分钟电话或文字就能对齐，比假设着做再返工省时间。

**推进第一版**：找最小闭环——能跑通核心路径、结果可以被人看到、看完之后有东西可以说"是/不是这个意思"。这次作业第一版就是"能接受 PDF 问一个问题、返回答案和来源"，其他都是后续。

**避免方向偏**：每做完一个可见的节点就同步一次，不等"全部做完再看"。方向偏是因为反馈周期太长，而不是因为没想清楚。

---

### 2. 关键节点前发现问题

**判断优先级**：先问"这个问题会不会让演示/交付直接挂掉"。会的话，不管多晚都要处理；不会的话，评估修的代价和收益，可以带着问题上线并在说明里注明。

**同步风险**：发现问题的第一时间同步，不等"我先试试能不能自己修好"——对方需要时间调整预期或协调资源，晚说一小时可能就晚了。同步内容：问题是什么、影响范围、我准备怎么处理、大概需要多久。

**自己的投入程度**：关键路径上的问题我会自己跟到底，不外包给"等明天再说"。这次 OCR 环境问题就是周末发现的，当天就切了降级方案跑通，没有留到第二天。

---

### 3. 沟通中出现分歧

**先搞清楚分歧的类型**：是对事实的分歧（可以用数据解决）、对优先级的分歧（可以用目标解决）、还是对实现路径的分歧（可以先跑一版再说）。大部分"分歧"其实是信息不对称，共享上下文后自然消解。

**保持推进节奏**：分歧不是暂停的理由。如果短时间对齐不了，我会提议"先按 A 方向跑一版，如果验证后有问题再改"，比一直讨论要快。

**避免反复拉扯**：决策要显式记下来（群里一句话或文档一条）；已经决定的事情不在新的上下文里重新讨论，有新信息再开新话题。如果有人比较着急，我会先认可他们的着急，再把"我们现在做的事情和着急的目标是一致的"说清楚。

---

### 4. 多件事情同时压过来

**排序方式**：先区分哪些有时间截止（演示、提交），哪些有阻塞关系（A 不完成 B 不能开始），哪些可以并行。时间截止的优先，阻塞链上游的优先，其他的排后或收尾时处理。

**AI / Agent 工具的使用**：
- 让 Claude Code 生成模块骨架和文档初稿，我来审阅逻辑；
- 并行任务中，把"不需要我判断"的机械性工作交给 AI，比如格式转换、注释补全、测试用例扩充建议；
- 文档排查（"有没有漏掉什么"）也可以让 AI 过一遍，但最后我自己再读一次。

**必须自己复查的结果**：评估报告的每道题答案（不只看通过率）、OCR 原始输出质量、任何对外交付的说明文档（客户会一字一字看）。

### 5. 讲两个真实例子

**例子一：SaaS 产品初期，在技术完备性和业务交付速度之间做方案决策**

负责一款 SaaS 产品从 0 到 1 的早期阶段，需要实现一套多租户数据处理系统。当时有两个方向：

- **借助公司中台**：有中台团队全程支持，系统对大规模数据的稳定性有保障，技术完备性高。代价是接入流程长、联调周期不可控，预估上线需要半年左右。
- **自己实现业务方案**：对准 MVP 核心功能，自行实现数据层，同时支持公有云和私有化部署混合场景，上线周期可以压到一个半月。

我做了一份两方案的对比分析，从上线时间、维护成本、扩展性、当前阶段适配度四个维度列出优劣，以及自研方案在稳定性上的具体风险和对应的止损手段（限流、告警、回滚预案）。带着这份分析去说服老板——不是说"我觉得自研更好"，而是说"产品早期客户规模小，稳定性风险可控，半年后再切中台才是合理时机；现在追求技术完备，等上线时窗口已经关了"。

老板认可后，自研方案在一个半月内上线，稳定运行三个月无规模性故障。后来数据量到位后有序迁移到中台，因为接口设计时预留了适配层，服务层不需要改动。

---

**例子二：多个需求同时冲突，保证都能按期上线**

某个迭代周期内，产品侧同时提了多个方向的需求，各方都认为自己优先级最高，但人力明显不够全部并行推进。

做法是和产品坐下来做一次排期评估：把每个需求的工作量、交付截止、依赖关系逐个拆开，看清楚哪些有硬截止、哪些可以分期、哪些开发阶段互不阻塞。有了这张图，优先级自然排出来了，不靠谁的立场来决定。

执行上做交叉开发和分期迭代——有硬截止的先做，主干开发和联调节点错开，大需求拆成核心功能先上、其余下个迭代，同时协调人力让不同模块并行推进。每周同步进度和风险，有阻塞当天说。

最终几个需求都按期上线，没有哪个因为"优先级之争"被完全搁置。核心逻辑是：**需求冲突不是资源问题，是信息问题**——把约束摆出来，排序自然清楚了。
